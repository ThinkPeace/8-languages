# 附录：八门语言的现实抽象哲学总表

> 建议在系统阅读前先浏览一遍，也建议在读完全书后再回来看第二遍。
> 第一遍把它当地图，第二遍把它当复盘表。

## 1. 一页总览：8 门语言分别把复杂性放在哪里

| 语言 | 默认气质 | 复杂性主要放置点 | 初读新代码时先看什么 | 最容易误读的地方 | 更常见的强项场景 |
| --- | --- | --- | --- | --- | --- |
| Python | 灵活、快、先跑通 | 运行时 + 团队规范 | 类型提示是否补齐、边界校验是否前置 | 代码短不等于约束强 | AI、脚本、数据处理、快速原型 |
| JavaScript | 事件驱动、互操作强 | 运行时 + 生态约定 | 异步边界、对象形状、运行时校验 | `async` 不等于天然并行和结构化 | Web 前后端、交互密集系统 |
| Java | 显式、稳定、长期治理 | 编译期 + 框架治理 | 类型边界、分层、异常翻译、线程模型 | 样板多不等于设计落后 | 企业后台、长期维护系统 |
| C++ | 高控制、贴近机器 | 开发者 + 工具链纪律 | 所有权、RAII、共享策略、性能热点 | 自由度高不等于默认安全 | 引擎、低延迟模块、系统软件 |
| Rust | 强约束、正确性前置 | 编译期 | 所有权、借用、错误类型、并发边界 | “难写”往往是在提前暴露真实问题 | 基础设施、可靠性敏感组件 |
| Go | 朴素、克制、工程统一 | 运行时 + 显式善后 | `context`、错误处理、资源释放、共享状态 | 简单不等于能力弱 | 云原生服务、高并发接口 |
| Swift | 安全、可预测、体验平衡 | 编译期 + ARC 协同 | 可选值、引用图、`defer`、任务边界 | ARC 不是 GC，释放语义更直接 | Apple 平台客户端、框架开发 |
| Kotlin | 现代化、务实、与 JVM 协同 | 编译期空安全 + 运行时生态 | 空值、协程、扩展函数、边界翻译 | 语法像 Java 不等于心智像 Java | Android、JVM 服务、渐进现代化项目 |

## 2. 如果你来自某个语言家族，最容易带错什么旧习惯

| 你来自哪里 | 你最容易带错的旧习惯 | 迁移时先提醒自己什么 |
| --- | --- | --- |
| Python / JavaScript | 先把东西跑起来，边界以后再补 | 看强约束语言时，不要把“啰嗦”误会成“设计落后” |
| Java / Kotlin | 习惯先设计接口和层次 | 看 Go、Python 时，先理解为什么它们宁可直接也不愿过度抽象 |
| Go | 习惯代码直接、显式善后 | 看 Rust、Swift 时，要接受编译器会更深入地追问关系边界 |
| Swift | 习惯对象生命周期较可预测 | 切回 GC 语言时，要主动补回对引用保留和监听器解绑的警惕 |
| C++ / Rust | 习惯先想清所有权和资源边界 | 切到托管语言时，不要为了“精确控制”过早把代码写复杂 |

## 3. 读陌生语言时，优先问这 7 个问题

1. 这门语言怎样建模现实对象？类型、可空、状态如何表达？
2. 它怎样组织对象关系？列表、映射、集合、不可变策略是什么？
3. 它怎样写业务规则？更偏早返回，还是更偏模式匹配和穷尽性？
4. 它怎样抽象行为？函数、接口、trait、protocol、委托分别承担什么角色？
5. 它怎样治理生命周期？GC、ARC、RAII、Ownership，谁在负责善后？
6. 它怎样组织并发？事件循环、线程、goroutine、协程、结构化并发哪个是主心智？
7. 它怎样面对失败？异常、错误返回、`Result`、边界翻译谁是默认姿势？

如果这 7 个问题你都有了初步答案，那么这门语言的“工程人格”已经大致显形了。

## 4. 按语言家族看，会更容易形成迁移桥

### 动态阵营：Python / JavaScript

关键词：灵活、快速表达、边界更多依赖运行时和工具链。

适合先回答的问题：

- 数据能不能先跑起来再收紧？
- 测试、lint、schema、类型提示补得够不够？
- 异步和事件驱动是否已经成为主要复杂度来源？

### 托管静态阵营：Java / Kotlin / Swift

关键词：边界更清晰、长期维护友好、语言与生态共同治理复杂性。

适合先回答的问题：

- 哪些约束可以直接写进类型和 API？
- 哪些复杂性更适合交给框架与平台惯例？
- 业务长期演化时，这套语义是否容易形成团队共识？

### 系统级阵营：C++ / Rust

关键词：资源与生命周期是一等问题，性能和控制力更直接。

适合先回答的问题：

- 这条链路是否真的需要更强控制和更低运行时代价？
- 团队是否有能力长期管理所有权、共享、unsafe 或工具链复杂度？
- 早期建模成本换来的后期稳定性，对当前业务值不值？

### 工程极简派：Go

关键词：少魔法、强一致性、通过克制换取协作效率。

适合先回答的问题：

- 这个团队是否更需要统一心智而不是更高表达力？
- 并发、错误和资源边界是否愿意显式写在代码里？
- 我是不是误把“简洁”理解成“功能不够”？

## 6. 最后一句总结

语言设计从来不是“谁更强”，而是“谁以什么方式抽象现实，并把代价放在什么位置”。

真正的横向学习能力，不是把这些表背下来，
而是以后遇到新语言时，你仍然知道该先看哪几列、先问哪几个问题。
