# 第七章：并发的艺术（多核时代的生存法则）

很多工程师第一次横向切语言时，最容易被并发打懵。

在 JavaScript 里，`await` 看起来顺理成章；到了 Go，别人开始和你谈 goroutine、channel、`context`；切到 Java，又是一整套线程池、Future、锁和调度；到 Rust，编译器还会冷冷地问你：这个值能不能在线程之间安全移动？

表面看，这是语法换了。其实不是。
真正变化的是：**不同语言对“同时发生的事情应该怎样组织”有完全不同的制度安排。**

## 本章目标

这一章要帮你建立的，不是某个库的熟练度，而是这三个判断力：

1. 分清“并发等待”“并行计算”“结构化任务管理”不是同一回事。
2. 看懂不同语言把并发复杂性放在了哪里：运行时、库、语言结构，还是类型系统。
3. 在迁移语言时，不再把旧语言里的并发直觉硬套到新语言上。

## 本章在“现实抽象链”中的位置

这一章处理抽象链的第七环：**单线流程 -> 多主体协作**。

前面的章节更多在讨论“一个流程怎么表达”；并发章节讨论的是：
**当多个流程同时存在、互相等待、彼此取消时，语言如何保持秩序。**

---

## 1. 先讲人话：并发不是“让程序更快”，而是“让多件事有秩序地同时发生”

初学并发时最常见的误区，是把所有问题都混成一句话：我要让程序快一点。

但真实工程里的并发，至少有三种完全不同的诉求：

1. **并发等待**：两个远程请求都很慢，我想同时等它们。
2. **并行计算**：一段 CPU 密集任务，我想让多个核一起算。
3. **任务治理**：子任务失败了，父任务要不要取消？超时怎么算？资源怎么回收？

语言在并发上的差异，往往不是谁能“同时做两件事”，而是谁更擅长处理其中哪一种诉求。

- JavaScript、Python 的强项常常是 I/O 并发。
- Java、C++ 的传统强项是线程和并行控制。
- Go 擅长把大量轻量任务组织起来。
- Swift、Kotlin、Rust 越来越强调结构化并发，让任务边界清楚。

所以，迁移一门语言的并发模型时，第一步不是背 API，而是先问：
**这门语言默认解决的是哪类“同时性问题”？**

## 2. 八语言同题：同时请求订单和库存，再合并结果

> 任务：订单详情页需要同时请求 `order` 和 `inventory`，两边都拿到之后再做合并。
> 这个例子只展示并发骨架；真实工程里你还需要加超时、取消和错误聚合。

### Python（asyncio）

```python
import asyncio

async def load_detail(order_id: str):
    order, inventory = await asyncio.gather(
        fetch_order(order_id),
        fetch_inventory(order_id),
    )
    return merge(order, inventory)
```

### JavaScript

```javascript
async function loadDetail(orderId) {
  const [order, inventory] = await Promise.all([
    fetchOrder(orderId),
    fetchInventory(orderId),
  ]);
  return merge(order, inventory);
}
```

### Java

```java
CompletableFuture<Detail> loadDetail(String orderId) {
    var orderFuture = fetchOrder(orderId);
    var inventoryFuture = fetchInventory(orderId);
    return orderFuture.thenCombine(inventoryFuture, this::merge);
}
```

### C++

```cpp
auto orderFuture = std::async(std::launch::async, fetch_order, order_id);
auto inventoryFuture = std::async(std::launch::async, fetch_inventory, order_id);
auto detail = merge(orderFuture.get(), inventoryFuture.get());
```

### Rust（Tokio）

```rust
let (order, inventory) = tokio::join!(
    fetch_order(order_id),
    fetch_inventory(order_id),
);
let detail = merge(order?, inventory?);
```

### Go

```go
orderCh := make(chan Order, 1)
inventoryCh := make(chan Inventory, 1)

go func() { orderCh <- fetchOrder(orderID) }()
go func() { inventoryCh <- fetchInventory(orderID) }()

detail := merge(<-orderCh, <-inventoryCh)
```

### Swift

```swift
func loadDetail(orderId: String) async throws -> Detail {
    async let order = fetchOrder(orderId: orderId)
    async let inventory = fetchInventory(orderId: orderId)
    return try await merge(order, inventory)
}
```

### Kotlin

```kotlin
suspend fun loadDetail(orderId: String): Detail = coroutineScope {
    val order = async { fetchOrder(orderId) }
    val inventory = async { fetchInventory(orderId) }
    merge(order.await(), inventory.await())
}
```

如果只看代码表面，你会觉得大家都差不多：无非是“同时发两个请求，再等结果”。
但真正的分野在代码背后：

- 谁负责调度这些任务？
- 任务失败时，兄弟任务会不会一起停？
- 这里的“同时”是事件循环上的同时，还是线程上的同时？
- 共享数据是靠约定避免竞争，还是靠语言机制限制竞争？

## 3. 同样的问题，复杂性到底被放到了哪里

| 路线 | 代表语言 | 复杂性主要放在哪 | 你换来的东西 | 典型代价 |
| --- | --- | --- | --- | --- |
| 事件循环 / 协程等待 | JavaScript、Python | 运行时调度和异步约定 | I/O 并发写法简洁 | 一旦阻塞主循环，系统表现会骤降 |
| 线程 / Future | Java、C++ | 线程池、锁、调度策略 | 表达力强，传统并发工具成熟 | 数据竞争、死锁、调优复杂 |
| goroutine / channel | Go | 任务生命周期和通信协议 | 高并发服务开发效率高 | goroutine 泄漏、取消和边界容易被忽略 |
| 结构化并发 | Swift、Kotlin、Rust（现代生态） | 语言或框架强制任务纳入父子关系 | 取消、异常传播、资源回收更清楚 | 早期设计要更讲边界 |

### 3.1 JavaScript / Python：先把 I/O 并发做好

这两门语言的一个共同优势，是让“同时等多个慢操作”这件事写起来相当自然。

但要注意，它们常见的默认并发并不等于“自动多核并行”。
很多时候你得到的是：

- I/O 等待期间不阻塞主流程
- 单线程事件循环或协作式调度下的高吞吐

这意味着，如果你把 CPU 重活直接塞进去，写法还是 `await`，结果却可能并不理想。

### 3.2 Java / C++：控制力强，但你得自己守住秩序

Java 和 C++ 在并发上的传统能力非常强，因为它们长期面对的就是线程、锁、池化、任务调度这些经典工程问题。

它们适合那些需要明确控制：

- 线程数
- 调度策略
- 锁粒度
- 共享内存性能

但代价很清楚：
一旦边界模糊，共享状态过多，复杂性会迅速上升。很多并发事故，根本不是因为 API 不够，而是因为“共享得太早、太广、太随意”。

### 3.3 Go：让并发便宜，但不替你定义边界

Go 的伟大之处，不只是 goroutine 很轻，而是它把并发写成了普通工程师也愿意天天使用的东西。

但这里藏着一个非常重要的迁移提醒：
**goroutine 便宜，不等于 goroutine 没成本。**

你仍然需要回答：

- 这个 goroutine 什么时候退出？
- 调用方取消了，它会不会还活着？
- 结果怎么回传？错误怎么汇总？

Go 不是没有结构，只是很多结构要靠你在工程层主动补上，比如 `context`、边界控制和单写者模型。

### 3.4 Swift / Kotlin / Rust：把“任务边界”变得更显式

现代并发设计的一条明显趋势，是不再鼓励任务像野草一样长出来。

Swift 的 `async let`、Kotlin 的 `coroutineScope`、Rust 生态里越来越流行的结构化思路，都在强调一件事：

- 子任务最好属于某个父任务
- 失败和取消要有明确传播路径
- 资源释放不应该和任务生命周期脱钩

这让代码在大型系统里更容易治理，因为任务不再只是“启动了”，而是“有来处，也有归处”。

## 4. 迁移提醒：并发里最危险的，不是 API 不熟，而是把旧语言的直觉带错地方

### 从 JavaScript / Python 切到 Go

你最容易带错的旧直觉是：
“既然我能很轻松地起一个任务，那它大概会像 `await` 一样自然结束。”

不会。

在 Go 里，启动一个 goroutine 只是开始，不是治理的完成。你必须主动考虑：

- 是否需要 `context` 取消
- 是否会阻塞在 channel 上
- 是否存在 goroutine 泄漏

### 从 Java / C++ 切到 async / coroutine 世界

你最容易带错的，是“只要我开了并发，就默认是线程并行”。

其实很多 `async` 写法表达的是调度协作，不一定意味着你真的拿到了多核并行能力。
所以，CPU 密集任务不要想当然地塞进事件循环或默认 dispatcher 里。

### 从 Go 切到 Swift / Kotlin

你会发现这些语言更强调“任务必须有父子关系”。
这不是多此一举，而是为了把取消、异常传播、资源善后变成默认秩序，而不是临时补丁。

### 从单线程心智切到共享内存并发

在 JavaScript 里，很多人对数据竞争没有身体感受；一到 Java、C++、Rust，这种感受会瞬间回来。

这时你要建立的新习惯是：

- 默认怀疑共享可变状态
- 默认优先消息传递或单写者
- 默认给超时和取消一个明确入口

## 5. 常见坑：多数并发 bug，不是“不会写并发”，而是“边界没有定义”

### 坑一：把 CPU 重活塞进 I/O 异步模型里

`async` 很优雅，但它解决的通常是等待问题，不一定是计算问题。
如果把大段 CPU 计算直接塞进去，结果往往是：接口看着现代，性能却并不现代。

### 坑二：只会启动任务，不会结束任务

很多服务线上出问题，不是因为并发太少，而是因为任务太多、太散、收不回来。

典型表现包括：

- goroutine 越积越多
- 线程池被长任务占满
- 请求已超时，但下游任务还在跑
- 父任务早结束了，子任务还在写日志和打数据库

### 坑三：一遇到共享数据就先上锁

锁不是不能用，但锁应该是最后确认边界之后的工具，而不是第一反应。
很多并发复杂度，其实在你决定“大家都来改同一个对象”那一刻就已经注定了。

### 坑四：忘记把取消、超时、重试当成并发设计的一部分

在真实系统里，并发从来不是孤立的“跑起来”。
它总是和这些问题绑在一起：

- 这个任务最长可以跑多久？
- 父请求取消后，子任务怎么办？
- 某个分支失败了，要不要让整体失败？

这些都不属于“异常情况”，而是并发设计的正文。

## 6. 什么时候偏向哪类语言：先看工作负载，再看团队治理能力

| 场景 | 更常见的合适选择 | 理由 |
| --- | --- | --- |
| I/O 密集、服务数量多、强调开发与部署效率 | Go、Java、Kotlin、JavaScript、Python | 并发等待能力和工程生态更重要 |
| 强调线程控制、成熟基础设施和大型团队治理 | Java | 线程池、监控、诊断工具成熟 |
| 追求极致性能、细粒度控制和底层并行能力 | C++、Rust | 更适合深入控制资源和共享状态 |
| 移动端 / 客户端，需要任务边界清楚、取消安全 | Swift、Kotlin | 结构化并发更符合 UI 与生命周期场景 |
| 高并发网络服务，希望模型直接、团队上手快 | Go | goroutine 与 channel 把并发门槛降得很低 |

选语言时，不要只问“这门语言并发强不强”。
更该问的是：

- 它默认把并发复杂性压在哪里？
- 我们团队有没有能力长期管理那部分复杂性？

---

## 回到贯穿主线：语言如何抽象现实

现实世界里的“同时发生”，从来都不是无边界的。
用户取消了请求，某个子任务失败了，库存已经更新但订单还没写库，消息已经发出但页面已经关闭——这些都说明，并发真正要处理的是**秩序**，不是热闹。

线程、协程、事件循环、goroutine、结构化并发，看上去是在提供不同工具；更深一层，它们是在回答同一个问题：
**多个动作同时发生时，系统如何避免失控。**

---

## 本章小结

并发模型不是语法风格，而是系统治理方式。

- JavaScript / Python 让 I/O 并发更轻松
- Java / C++ 给你强控制力，也把更多秩序责任交给你
- Go 让并发普及化，但边界仍要你自己设计
- Swift / Kotlin / Rust 越来越强调任务必须有结构、有父子关系

当你能先看懂一门语言怎样组织“同时发生的事”，你才真正开始掌握它的并发哲学，而不只是学会几个 `async` 关键字。
