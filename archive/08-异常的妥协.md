# 第八章：异常的妥协（面对失败的态度）

很多工程师第一次从异常语言切到 Go，会觉得 `if err != nil` 像在抄经；第一次从 Go 切到 Rust，又会觉得每一步都在被提醒“失败是常态”；切回 Python、Java、Kotlin 时，又会觉得主流程终于清爽了，但心里隐约不安：有些错误是不是被藏起来了？

这种不适感非常正常。因为错误处理从来不只是写法问题，它背后问的是更严肃的事：

- 谁有责任看见失败？
- 失败是靠语法跳转，还是靠返回值显式传播？
- 系统怎样保留上下文，而不是只留下一个模糊的字符串？

也正因为如此，错误模型常常决定了一支团队的代码气质：是倾向把主流程写得干净，还是把失败路径写得明白；是把纪律更多交给人，还是更多交给语言。

## 本章目标

这一章要帮你建立三种判断：

1. 看懂异常、返回值、`Result` 这些机制分别把复杂性放在了哪里。
2. 知道从一种错误文化迁移到另一种时，最容易带错的习惯是什么。
3. 理解成熟系统真正关心的不是“语法选边”，而是错误分层、上下文保留和边界翻译。

## 本章在“现实抽象链”中的位置

这一章处理抽象链的第八环：**理想流程 -> 失败世界**。

前面的章节讨论的是系统如何表达正确路径；到了这里，语言开始回答另一个更接近现实的问题：
**事情没按预期发生时，系统怎样不崩、怎样说清、怎样恢复。**

---

## 1. 先讲人话：错误处理真正处理的，不是“报错”，而是“失败秩序”

很多人把错误处理理解成“出了问题怎么打印日志”。这是不够的。

一个成熟系统至少要把失败世界里的三件事安排明白：

1. **表达**：失败用什么形式出现？异常、错误值、`Result`，还是别的？
2. **传播**：失败怎样往上游传？会不会被中途吞掉？
3. **翻译**：底层错误到了用户/API 边界，要不要换成稳定的业务错误？

所以，错误模型不是附属品。它决定的是：
当事情出错时，系统是清楚地失控，还是糊涂地失控。

## 2. 八语言同题：读取优惠规则，并把底层失败包装成上层可理解错误

> 任务：读取 `coupon_rules.json`，解析规则，并把底层 I/O 或解析错误包装成上层的“加载优惠规则失败”。

### Python

```python
def load_coupon_rules(path: str):
    try:
        with open(path, "r", encoding="utf-8") as f:
            return parse_rules(f.read())
    except OSError as e:
        raise RuntimeError("load coupon rules failed") from e
```

### JavaScript

```javascript
async function loadCouponRules(path) {
  try {
    const text = await fs.promises.readFile(path, "utf8");
    return parseRules(text);
  } catch (e) {
    throw new Error(`load coupon rules failed: ${e.message}`);
  }
}
```

### Java

```java
CouponRules loadCouponRules(Path path) {
    try {
        var text = Files.readString(path);
        return parseRules(text);
    } catch (IOException e) {
        throw new CouponRuleException("load coupon rules failed", e);
    }
}
```

### C++

```cpp
CouponRules load_coupon_rules(const std::string& path) {
    try {
        std::ifstream in(path);
        if (!in) throw std::runtime_error("open failed");
        return parse_rules(in);
    } catch (const std::exception& e) {
        throw std::runtime_error(std::string("load coupon rules failed: ") + e.what());
    }
}
```

### Rust

```rust
fn load_coupon_rules(path: &str) -> Result<CouponRules, AppError> {
    let text = std::fs::read_to_string(path).map_err(AppError::Io)?;
    parse_rules(&text).map_err(AppError::Parse)
}
```

### Go

```go
func loadCouponRules(path string) (CouponRules, error) {
    data, err := os.ReadFile(path)
    if err != nil {
        return CouponRules{}, fmt.Errorf("load coupon rules failed: %w", err)
    }
    return parseRules(data)
}
```

### Swift

```swift
func loadCouponRules(path: String) throws -> CouponRules {
    let text = try String(contentsOfFile: path, encoding: .utf8)
    return try parseRules(text)
}
```

### Kotlin

```kotlin
fun loadCouponRules(path: String): CouponRules {
    return try {
        val text = java.io.File(path).readText()
        parseRules(text)
    } catch (e: Exception) {
        throw CouponRuleException("load coupon rules failed", e)
    }
}
```

你会看到三种非常不同的气质：

- 有的语言让正常路径最干净，失败通过异常“跳出去”。
- 有的语言让每一步都显式承认失败可能发生。
- 有的语言试图在两者之间找平衡。

所以，错误模型的差异从来不只是“写几行”的差异，而是：
**语言希望你在多大程度上正视失败。**

## 3. 同样的问题，复杂性到底被放到了哪里

| 路线 | 代表语言 | 复杂性主要放在哪 | 你换来的东西 | 典型代价 |
| --- | --- | --- | --- | --- |
| 异常主导 | Python、JavaScript、Java、Kotlin、C++、Swift | 控制流跳转和团队纪律 | 正常路径更干净 | 错误路径可能被忽视或过度分散 |
| 显式错误返回 | Go | 调用方代码和边界处理 | 失败路径一目了然 | 样板代码增多，容易机械重复 |
| 类型化失败 | Rust（Swift 的一部分场景也接近） | 类型系统与接口设计 | 错误被纳入契约 | 早期建模和封装成本更高 |

### 3.1 异常路线：让主流程干净，但要求团队守纪律

异常的最大优点，是把“正常业务路径”保留下来。
你读一段代码时，眼睛不会被层层 `if err != nil` 打断，这对表达业务意图非常有利。

但异常也会带来一个经典风险：
错误路径不再全部写在眼前，而是散落在调用栈和 catch 边界里。

所以异常文化成熟的团队，往往会特别强调：

- 哪些异常要就地处理，哪些要继续上抛
- 哪些边界负责统一翻译异常
- 日志到底应该记在哪里，避免重复记录

如果没有这些纪律，异常代码会很快变成“平时很干净，出事时很模糊”。

### 3.2 Go：失败是正文，不是脚注

Go 的错误观很朴素：
失败不是少见情况，所以应该被写出来。

这让 Go 代码有一种很强的可见性。你一眼就能看出：

- 哪一步会失败
- 失败后在哪返回
- 上下文有没有被补充进去

但 Go 的问题也很真实：
如果团队只会机械地复制 `if err != nil`，而不会抽象边界、补充上下文、区分错误类型，那么代码会变得啰嗦却不清楚。

所以，Go 的关键不是“显式就好”，而是“显式之后要有层次”。

### 3.3 Rust：错误是契约的一部分

Rust 的一个根本立场是：
如果失败是可预期的，那它就应该出现在类型里。

这会迫使你更早回答几个问题：

- 这一步失败属于哪一类？
- 调用者是否必须处理？
- 哪些错误可以统一上浮，哪些要就地转换？

所以很多人第一次写 Rust 会觉得“为什么到处都要管错误”。
其实不是 Rust 特别悲观，而是它不愿意帮你假装失败很少发生。

## 4. 真正成熟的系统，不先争论语法，而先给失败分层

一本写给工程实践的书，不能停在“异常 vs 返回值”的口水战上。
真正决定系统质量的，是你是否把失败世界分层了。

| 错误层次 | 例子 | 应对方式 |
| --- | --- | --- |
| 领域错误 | 库存不足、优惠券已过期 | 返回稳定业务码，给用户可理解提示 |
| 技术错误 | 数据库超时、磁盘损坏、网络抖动 | 保留 cause，做监控、重试、降级 |
| 边界错误 | 对外 API 的状态码、错误协议 | 统一翻译，不泄漏内部实现细节 |

这张表比任何语法争论都重要。因为再优雅的错误机制，如果没有分层，最后都会落到同一种混乱：

- 用户看到了不该看到的底层错误
- 监控系统收不到可聚合的错误码
- 业务错误被当技术错误重试
- 技术错误被当业务错误吞掉

## 5. 迁移提醒：错误文化切换时，旧语言的“舒适动作”最危险

### 从异常语言切到 Go / Rust

你最容易带错的习惯是：
以为错误只要最终有人兜底就行。

但在 Go / Rust 里，更好的习惯是：

- 在出错的地方补充上下文
- 不要只是“返回错误”，要返回可判断的错误
- 早点决定哪些错误要被翻译成稳定领域语义

### 从 Go / Rust 切到异常语言

你最容易丢掉的，是“失败路径的可见性”。

这时要主动补回三件事：

- 关键边界统一 catch 并翻译
- 不要一边记录日志一边继续抛出，制造重复噪音
- 把异常层次和业务错误码文档化

### JavaScript / Python 的额外提醒

在异步环境里，失败不一定发生在你眼前这行代码上。
Promise rejection、后台任务失败、事件回调异常，常常让错误路径比同步代码更分散。

所以在这两门语言里，除了写 `try/catch`，还要格外重视：

- 异步任务的统一兜底
- 未处理异常的监控入口
- 错误上下文不要只留一段字符串

## 6. 常见坑：多数错误处理事故，不是“不会捕获”，而是“失败语义不清”

### 坑一：记录日志后继续原样抛出

这会制造一类非常常见的生产事故：
同一个错误在三层日志里被记了三遍，但每层都没补充新的上下文。

结果是日志很多，信息却不多。

### 坑二：只传字符串，不保留 cause

一旦丢了底层 cause，你很快就会陷入“上层知道失败了，但不知道为什么失败”的状态。
这对排障和监控聚合都很糟糕。

### 坑三：把所有错误都塞进一个巨大 `AppError`

统一错误类型本身没问题，问题在于如果它失去了区分能力，调用者就无法根据错误类别做正确动作：

- 是提示用户重试？
- 是自动重试？
- 是报警？
- 还是直接失败退出？

### 坑四：业务错误和技术错误混在一起处理

“库存不足”通常不该自动重试；“数据库连接超时”通常不该直接返回给用户看。
如果这两类错误混在一起，再好的语法也救不了架构混乱。

## 7. 什么时候偏向哪类语言：错误模型也服务于场景，而不是服务于审美

| 场景 | 更常见的合适选择 | 原因 |
| --- | --- | --- |
| 业务流程复杂，希望主路径保持清晰 | Java、Kotlin、Python、Swift | 异常让业务表达更顺 | 
| 强调可读性、显式性、团队统一风格 | Go | 错误处理路径始终在眼前 |
| 希望把失败契约做得很强，提前约束调用方 | Rust | 类型化错误适合做可靠性治理 |
| 生态历史包袱重、需要兼容既有体系 | Java、Kotlin、JavaScript | 错误机制可渐进演进 |

关键不是“哪种最好”，而是：
**你的团队更擅长在什么地方承担失败复杂性。**

- 如果团队善于制度化分层和边界治理，异常会很高效。
- 如果团队更重视路径显式和一致风格，Go 会很舒服。
- 如果你希望大量错误在接口层就被强制看见，Rust 会更有力量。

## 回到贯穿主线：语言如何抽象现实

现实世界里的失败，不会因为语言更优雅就减少。
用户会输错参数，磁盘会损坏，网络会超时，第三方服务会不稳定。

错误模型真正决定的，是系统面对这些失败时的姿态：

- 是把失败隐藏在控制流后面，换来更干净的主路径
- 还是把失败显式写在每一步里，换来更强的可见性
- 还是把失败纳入类型契约，换来更强的前置约束

所以，错误处理不是附属语法，而是一门语言对“不确定性”的治理立场。

---

## 本章小结

错误处理决定了一门语言的失败素养。

- 异常让主流程更自然，但要求团队建立严格的边界纪律
- Go 把失败写到台前，代价是样板和抽象都要做得克制而清楚
- Rust 把失败纳入契约，前期更重，后期更稳

真正成熟的工程师，不会停留在“我更喜欢哪种写法”，而会继续追问：
这套错误制度，是否能让我们的系统在失败来临时仍然说得清、收得住、修得动。
