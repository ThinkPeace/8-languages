# 写在前面：为什么是这 8 门语言

你很可能经历过这样的切换：

- 上午在 Python 里改一个数据脚本，觉得“先跑通再说”完全合理
- 中午去看 Java 服务，发现每个字段、每个接口都写得很满，像在提前签合同
- 下午碰到 Go，代码看着朴素，却处处在提醒你“别耍聪明”
- 晚上读 Rust，又发现编译器像一个极其严格、但也极其负责的搭档

很多人以为，跨语言学习最难的是语法。
其实不是。
真正难的是：你脑子里那套“什么可以先省略、什么必须先说清”的默认习惯，会在不同语言里不断失效。

所以这本书不打算把 8 门语言讲成 8 本语法手册。
它真正想做的，是帮你建立一套更稳定的判断力：
当你从一门语言走到另一门语言时，你知道自己到底换掉了什么。

## 这本书写给谁

这本书最适合下面这类读者：

- 已经会至少一门主流语言，但一换语言就容易“会看不会写”
- 工作里需要读、改、评审多种语言代码
- 不满足于背 API，希望看懂语言背后的工程取舍
- 想把“学语言”升级成“学抽象、学设计、学迁移”

它不太适合作为你的第一本编程入门书。
因为这本书默认你已经写过程序，已经吃过一些亏：空值的亏、并发的亏、类型不清的亏、重构崩掉的亏。
你带着这些经验来读，它会更有用。

## 这本书到底想解决什么问题

很多跨语言资料的问题是：

- 过于像词典，只告诉你“这门语言里 `for` 怎么写”
- 过于像宣传稿，只强调“某门语言多先进”
- 过于像碎片博客，每篇都讲一个点，但拼不成完整认知

而真正的工作现场并不是这样。
你面对的是一个具体问题：

- 订单到底怎么建模？
- 缺失值到底该怎么表示？
- 并发到底由语言兜底，还是由团队纪律兜底？
- 失败到底是抛异常、返回错误，还是被类型系统提前拦住？

所以本书采用一个更朴素、也更耐用的问法：
**同一个现实问题，8 门语言各自会怎么回答？**

只要这个问题问稳了，语法差异反而会慢慢退到次要位置。

## 为什么偏偏是这 8 门语言

不是因为它们“最流行”或者“最先进”，而是因为它们刚好覆盖了现代软件工程里几条非常典型的路线。

- Python：高灵活度，强调开发速度与生态效率
- JavaScript：围绕 Web 世界长出来，事件与兼容性意识非常强
- Java：强调类型清晰、团队协作、长期维护
- C++：给你极高控制权，也把复杂度如实交还给你
- Rust：把大量正确性问题前移到编译期
- Go：刻意克制表达力，换取工程一致性与可读性
- Swift：重视安全、可预测性和开发体验的平衡
- Kotlin：在现实 JVM 世界中尽量把语法和空值问题做现代化改良

把这 8 门语言放在一起看，你会更容易明白：
**语言之间真正的差别，不是功能多少，而是复杂性放在了哪里。**

有的语言让你先快起来，出了问题再补。
有的语言逼你先把问题说清，再允许你继续写。
还有的语言给你高度自由，但同时也把后果一并交给你承担。

## 本书不按语言分册，而按问题展开

如果按语言写，最容易变成：Python 一章、Java 一章、Go 一章……
这样当然整齐，但读者往往会掉进一个陷阱：
每次都在语言内部打转，却看不清它们之间真正可比较的地方。

所以这本书反过来组织：

- 先看共同问题：数据、容器、控制流、函数、抽象
- 再看分野问题：内存、并发、错误处理
- 最后总结：如何看懂第九门语言

这样安排的好处是，你不是在记“某语言有哪些特性”，
而是在练一种更接近工程现场的能力：
**面对同一类问题，迅速判断一门语言会把风险放在哪、把便利给到哪。**

## 贯穿全书的一条主线

全书反复使用同一条观察链路：

1. 现实里真正的问题是什么
2. 程序里要把它建成什么模型
3. 语言用什么约束来防止你胡来
4. 运行时或编译器怎样落实这些约束
5. 这样做换来了什么，又牺牲了什么
6. 在团队和系统里，最后如何验证它是否真的值得

你可以把它理解成一套固定问法。
以后遇到第九门、第十门语言时，也仍然能用。

## 为什么全书围绕同一个业务背景

本书大量例子围绕同一条主线：订单系统。

原因很简单。
如果每章都换一个场景，这一章讲电商、下一章讲游戏、再下一章讲编译器，读者会不断被业务细节带偏。
看起来信息很多，真正留下的却很少。

而如果始终盯着同一类对象——订单、用户、支付、库存、异常、并发——你就能更清楚地看到：
同一份现实，在不同语言里到底被切成了什么样子。

这比堆很多花哨例子更有教学价值。

## 你应该怎样读这本书

有三种读法。

第一种是顺着读。
如果你想系统建立跨语言坐标系，就从前言一直读到“看懂第九门语言”。这条路线最完整。

第二种是对照读。
如果你已经会一门语言，比如 Python，那么读每章时就不断问自己：
“如果这段逻辑写在我熟悉的语言里，我会默认怎么做？而这里为什么不一样？”
这种读法最能产生迁移感。

第三种是带着问题读。
如果你最近正被空值、并发、错误处理、内存边界这些问题折磨，就先跳到对应章节，再回头补前面的共性部分。

无论哪种读法，我都建议你少问一句“这一门语言怎么写”，多问一句：
**“它为什么鼓励我这样写？”**

## 读完之后，你应该获得什么

理想情况下，读完这本书后，你会多出三种能力。

第一，看到新语言时，不会立刻陷入语法焦虑。
你会先找它的默认假设：它相信程序员多少？它提防程序员多少？它把哪些错误前移了？

第二，做技术选型时，不会只看热度和招聘市场。
你会更关心：团队规模、业务类型、故障成本、交付速度，到底和哪种语言哲学更匹配。

第三，做跨语言协作时，不会只停留在“风格不同”。
你会知道那其实常常不是风格，而是两套不同的风险治理方式。

## 最后说一句

这本书不会承诺你“学完就精通 8 门语言”。
那不现实，也没必要。

它真正希望给你的，是一种更长期的东西：
当语法、框架、流行趋势不断变化时，你仍然能抓住那些不太变的部分——
现实问题、抽象方式、约束机制、工程代价。

语法会过时，框架会更替，甚至语言本身也会起落。
但一个程序员如果能持续看懂“复杂性被放在了哪里”，他就不会轻易在语言切换中迷路。

如果这本书能帮你建立起这套坐标系，它的目的就达到了。
