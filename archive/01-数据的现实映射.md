# 第一章：数据的现实映射（变量与类型系统）

## 先从一个很小、但很常见的事故开始

订单系统收到一条新数据：

- 订单号：`A1001`
- 金额：`-99`
- 是否支付：`true`
- 优惠码：没有

这时程序要立刻回答三个问题：

1. `-99` 这种金额是不是非法数据？
2. “没有优惠码”到底该记成 `null`、空字符串，还是根本不传？
3. 这些规则应该靠程序员自觉，还是靠语言、编译器、运行时帮你兜住？

这就是类型系统真正登场的地方。

很多人一提“类型”，脑子里先出现的是 `int`、`string`、`bool`。
但在工程里，类型从来不只是语法标签。
它更像是一份边界说明书：
**什么值可以进来，什么状态不该存在，什么错误应该尽早暴露。**

## 本章你要带走什么

读完这一章，你应该能抓住四件事：

- 为什么“定义一个字段”其实是在裁剪现实
- 为什么同一个订单模型，在 8 门语言里会长得很不一样
- 动态与静态的真正分歧，为什么是“风险放在哪个阶段”
- 为什么空值问题几乎总是跨语言迁移时最先绊倒人的地方

## 本章在全书主线中的位置

全书的主线是：现实 -> 模型 -> 约束 -> 机制 -> 代价 -> 反馈。

这一章讨论的是最前面的两步：
**你先怎样看待现实对象，再怎样把它变成程序里的数据模型。**

一旦这一步做得草率，后面所有章节都会跟着变难：
容器会乱，控制流会绕，异常会飘，并发会更危险。

## 1. 写代码之前，你先在删减现实

现实世界里的“订单”远比程序里的 `Order` 复杂。

一个真实订单可能还带着：

- 下单时间
- 支付渠道
- 税费与运费
- 退款状态
- 风控标记
- 业务来源
- 审计痕迹

但我们不会一上来把所有现实都搬进代码。
程序建模的第一步，永远是裁剪。

比如，在本章里我们只保留四个字段：

- `orderId`：订单身份
- `amount`：订单金额
- `paid`：是否已支付
- `couponCode`：是否用了优惠码

这个动作看似普通，其实已经包含了三个判断：

1. 哪些信息是当前问题必须关心的
2. 哪些信息虽然真实存在，但此刻先不建模
3. 哪些信息必须被约束，否则后面一定出事

所以，所谓“定义数据结构”，本质上不是在堆字段。
而是在说：
**对当前问题而言，现实里哪些差别值得被保留下来。**

## 2. 一个成熟模型，至少要回答这三个问题

当我们说“把订单建成一个类型”，至少要回答下面三个问题。

| 现实中的问题 | 代码里的表达 | 如果没说清，会发生什么 |
| --- | --- | --- |
| 订单靠什么被唯一识别？ | `orderId: String` | 去重、追踪、排错都会混乱 |
| 金额允许哪些值？ | `amount: number` + 非负约束 | 折扣、退款、结算逻辑会被污染 |
| 优惠码可以不存在吗？ | 可空/可选类型 | 到处写防空判断，或者线上空指针 |

这里尤其要注意一个初学者常忽略、但有经验的人很在意的点：
**字段类型只解决“长得像不像”，不自动解决“业务上合不合法”。**

比如 `-99` 当然是数字，语法上没错。
但业务上它可能完全不合法。
所以“类型”与“校验”通常要配合出现。

## 3. 先别急着看语法，先看 8 门语言各自把责任放在哪

下面我们用同一个小任务做对照：

> 定义一个订单对象，并拒绝负数金额。

读这些代码时，先不要急着比较谁更短、谁更优雅。
先看三件事：

1. 非法金额是在创建时被拦住，还是之后靠调用 `validate()`？
2. “优惠码可不存在”是语言内建能力，还是约定俗成？
3. 对象创建之后，字段还能不能随手被改掉？

### Python

```python
from dataclasses import dataclass
from typing import Optional

@dataclass(frozen=True)
class Order:
    order_id: str
    amount: float
    paid: bool
    coupon_code: Optional[str] = None

    def __post_init__(self) -> None:
        if self.amount < 0:
            raise ValueError("amount must be >= 0")
```

Python 可以写得很快，也允许你先把模型搭起来。
但 `Optional[str]` 更多是在表达意图；真正是否严格执行，往往还要靠测试、类型检查工具和团队习惯共同完成。

### JavaScript

```javascript
class Order {
  constructor({ orderId, amount, paid, couponCode = null }) {
    if (amount < 0) {
      throw new Error("amount must be >= 0");
    }

    this.orderId = orderId;
    this.amount = amount;
    this.paid = paid;
    this.couponCode = couponCode;
  }
}
```

JavaScript 的重点通常是灵活与互操作。
你能很快把对象组织出来，但很多边界默认不是语言替你守住，而是靠运行时约束、测试和更上层的工具体系来补强。

### Java

```java
public record Order(String orderId, double amount, boolean paid, String couponCode) {
    public Order {
        if (amount < 0) {
            throw new IllegalArgumentException("amount must be >= 0");
        }
    }
}
```

Java 倾向于让模型清晰、接口明确。
`record` 让“这是一份稳定的数据”这件事表达得很直接，但 Java 本身不会像 Kotlin 那样天然把空值问题压得很严，所以团队往往还会引入额外规范或注解体系。

### C++

```cpp
#include <optional>
#include <stdexcept>
#include <string>

struct Order {
    std::string order_id;
    double amount;
    bool paid;
    std::optional<std::string> coupon_code;

    Order(std::string id, double value, bool is_paid,
          std::optional<std::string> coupon = std::nullopt)
        : order_id(std::move(id)), amount(value), paid(is_paid), coupon_code(std::move(coupon)) {
        if (amount < 0) {
            throw std::invalid_argument("amount must be >= 0");
        }
    }
};
```

C++ 给你很强的表达力与控制权。
`std::optional` 很明确，但语言的整体态度仍然是：工具给你，后果也交给你。
它很强，却不太替你操心。

### Rust

```rust
#[derive(Debug)]
struct Order {
    order_id: String,
    amount: f64,
    paid: bool,
    coupon_code: Option<String>,
}

impl Order {
    fn new(order_id: String, amount: f64, paid: bool, coupon_code: Option<String>) -> Result<Self, String> {
        if amount < 0.0 {
            return Err("amount must be >= 0".into());
        }

        Ok(Self {
            order_id,
            amount,
            paid,
            coupon_code,
        })
    }
}
```

Rust 的典型做法，是尽量把“非法状态”挡在对象建立之前。
`Option` 和 `Result` 不是附属工具，而是它世界观的中心：
缺失要显式，失败也要显式。

### Go

```go
package order

import "errors"

type Order struct {
    OrderID    string
    Amount     float64
    Paid       bool
    CouponCode *string
}

func NewOrder(orderID string, amount float64, paid bool, couponCode *string) (Order, error) {
    if amount < 0 {
        return Order{}, errors.New("amount must be >= 0")
    }

    return Order{
        OrderID:    orderID,
        Amount:     amount,
        Paid:       paid,
        CouponCode: couponCode,
    }, nil
}
```

Go 不喜欢把简单问题说得过分花哨。
它的风格很像在说：规则写清楚，错误老老实实返回，团队成员都看得懂，比语法炫技更重要。

### Swift

```swift
enum ValidationError: Error {
    case invalidAmount
}

struct Order {
    let orderId: String
    let amount: Double
    let paid: Bool
    let couponCode: String?

    init(orderId: String, amount: Double, paid: Bool, couponCode: String? = nil) throws {
        guard amount >= 0 else {
            throw ValidationError.invalidAmount
        }

        self.orderId = orderId
        self.amount = amount
        self.paid = paid
        self.couponCode = couponCode
    }
}
```

Swift 很擅长把“安全”做成你日常写代码时自然会碰到的东西。
可选值、构造期校验、值语义，这些组合起来会让很多移动端常见事故提前暴露。

### Kotlin

```kotlin
data class Order(
    val orderId: String,
    val amount: Double,
    val paid: Boolean,
    val couponCode: String?
) {
    init {
        require(amount >= 0) { "amount must be >= 0" }
    }
}
```

Kotlin 的感觉像是：承认 JVM 世界的现实，但尽量把那些常见痛点处理得更顺手。
尤其是空值系统，会明显改变你写业务代码时的默认习惯。

## 4. 把 8 段代码放在一起，你应该先看出什么

如果只看表面，差异好像是语法差异。
但如果看责任分配，差异就更清楚了。

| 语言 | 非法金额通常在哪拦 | “没有优惠码”怎么表达 | 语言替你挡错的力度 |
| --- | --- | --- | --- |
| Python | 运行时，常靠构造或显式校验 | `Optional[str]` / `None` | 中等，更多依赖工具链 |
| JavaScript | 运行时校验 | `null` / `undefined` | 偏弱，灵活度高 |
| Java | 构造期或框架层校验 | `null` | 中等，类型清楚但空值仍危险 |
| C++ | 构造期或调用约定 | `std::optional<T>` | 中等到强，但控制权也更重 |
| Rust | 构造期返回 `Result` | `Option<T>` | 很强，鼓励显式处理 |
| Go | 工厂函数返回 `error` | 指针或零值策略 | 中等，强调明确而朴素 |
| Swift | 初始化期 `throw` | `T?` | 强，空值是语言一等公民 |
| Kotlin | 初始化期 `require` | `T?` | 强，日常开发就会受到约束 |

同样是“定义一个订单”，有的语言更像在说：
“先写出来，出错时再处理。”

有的语言更像在说：
“先把边界说清，不然别往下走。”

这就是类型系统最重要的分歧。
不是语法的分歧，而是风险治理方式的分歧。

## 5. 动态与静态，真正争论的不是高级不高级

关于动态类型和静态类型，最常见的误解是把它讲成“先进 vs 落后”。
这几乎总是无效争论。

更有用的问法是：
**这个团队，希望把错误尽量暴露在什么时候？**

举一个很实际的例子。
你把 `amount` 字段误写成了 `amout`。

- 在 Python、JavaScript 一类语言里，这种错误常常要等代码跑到那一行、测试覆盖到那一条路径，问题才会露出来
- 在 Java、Kotlin、Rust、Swift 这类约束更强的环境里，它更可能在编译期或更早的检查阶段就被指出来

这并不意味着动态语言“不好”。
如果你正处在需求快速变化期，动态语言带来的低摩擦感非常珍贵。
但一旦系统进入多人协作、长期维护、跨团队交接阶段，提前说清楚边界的收益会越来越大。

所以动态与静态的本质，不是表达力高低。
而是：

- 动态语言把更多判断放到运行期
- 静态语言把更多判断提前到编译期

换句话说，区别在于：
**你把不确定性留给未来的自己，还是今天就处理掉一部分。**

## 6. 空值问题，是跨语言迁移时最容易摔跤的地方

很多线上事故，其实都不是复杂算法造成的。
而是一个很朴素的问题：
某个值原本以为一定有，结果它没有。

比如优惠码 `couponCode`。
如果它可能不存在，那么下面这件事就必须被回答：

- 能不能直接调用它的方法？
- 调用前是不是必须判断？
- 没有优惠码时，业务上到底表示“未填写”、还是“明确没有”？

不同语言在这件事上的态度非常不一样。

- Kotlin、Swift、Rust 会强迫你承认“缺失”是一种正式状态
- Java 可以借助 `Optional` 和注解改善，但语言历史包袱让它不可能天然那么彻底
- Python、JavaScript 往往更依赖约定、lint、测试、schema 校验来补足

这也是为什么很多人从 Python/JavaScript 转到 Kotlin/Swift/Rust 时，第一感觉是“怎么这么麻烦”。
因为原来靠经验脑补的东西，现在必须被写出来。

反过来，从 Kotlin/Swift/Rust 回到 Python/JavaScript 的人，也很容易吃亏。
因为编译器不再替你盯着这些缺失状态了，你得自己把纪律补回去。

## 7. 给有经验读者的四个实战提醒

到这里，我们可以先不谈语言立场，只谈更稳妥的工程习惯。

### 第一，尽量把校验放在对象建立附近

如果一个订单从一开始就不该允许负数金额，那最好在创建时就拦住。
越晚校验，非法状态在系统里游荡的时间就越长。

### 第二，尽早定义“缺失”到底是什么意思

`null`、空字符串、默认值、未传字段，这四者在业务上经常不是一回事。
你如果一开始没分清，后面查询、序列化、接口兼容都会乱。

### 第三，金额示例里用浮点数只是为了聚焦主题

本章为了让 8 门语言更容易横向对照，使用了 `float` / `double`。
但在真正的金融、结算、账务场景里，金额通常应该使用定点数、十进制类型，或最小货币单位整数。

这条提醒很重要。
因为“示例可读性”和“生产正确性”有时不是同一个最优点。
一本负责任的技术书应该明确告诉你这件事。

### 第四，跨团队边界上的类型要更保守

函数内部临时偷一点懒，代价也许有限。
但一旦进入：

- API 请求与响应
- 消息队列事件
- 数据库存储模型
- 多团队共享的 SDK

类型与空值策略就应该尽量明确。
因为边界一模糊，沟通成本会远高于多写几行定义。

## 8. 迁移提醒：别只学写法，要学默认假设

如果你现在熟悉的是 Python 或 JavaScript，读强约束语言时要特别注意：

- 不要把“编译器真烦”理解成坏事
- 它烦你的地方，往往正是线上最爱出事故的地方
- 你以前靠经验兜住的边界，现在被要求显式说出来了

如果你熟悉的是 Java、Kotlin、Swift、Rust，再去看动态语言，也要调整预期：

- 很多约束不会天然存在
- 你需要把 schema、lint、测试、运行时校验重新补齐
- “写得快”通常是真的，但“以后也稳”要靠额外工程措施换来

跨语言迁移最忌讳的一件事，就是把自己在 A 语言里的熟练，误当成放之四海而皆准的常识。
很多所谓“这门语言设计得不好”，本质上只是：
你还在用上一门语言的默认假设理解它。

## 回到全书主线：类型是第一道抽象边界

到这里我们就能回到本书那条总主线了。

现实中的订单，进入程序之后，并不会自动变成一个“正确的数据结构”。
中间要经过一连串判断：

- 哪些信息值得保留
- 哪些状态必须被允许出现
- 哪些状态必须被明确禁止
- 哪些错误要尽早暴露

这就是类型系统的意义。

它不是在帮你把代码写得更像数学题。
它是在帮你决定：
**哪些混乱可以被带进系统，哪些混乱必须被挡在门外。**

## 本章小结

这一章最重要的，不是记住 8 门语言的语法写法。
而是记住下面三句话：

- 数据模型首先是对现实的裁剪，不是字段堆砌
- 类型系统的差异，本质上是风险治理方式的差异
- 空值与非法状态，越早被说清，后面的系统就越稳

从下一章开始，我们会继续往前走：
当对象不再是单个值，而要形成列表、映射、集合时，语言又会怎样继续暴露它的世界观。
