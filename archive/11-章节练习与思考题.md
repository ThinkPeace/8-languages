# 章节练习与思考题

> 建议每章至少完成 1 道实现题、1 道迁移题、1 道评审题、1 道变更实验题。
> 这份附录不是为了考语法记忆，而是为了把本书的方法论真正练成跨语言判断力。

## 使用方式

你可以把每章练习分成四类：

- `实现题`：把同一业务题在不同语言里落地，训练横向对照能力
- `迁移题`：从你熟悉的语言切到另一门语言，找出默认假设哪里变了
- `评审题`：站在 reviewer 视角看边界、坏味道和故障风险
- `变更实验`：人为加入新需求，看原有设计在不同语言里如何承压

---

## 第 1 章：数据的现实映射

- `实现题`：把一个只用 `string / number / null` 的订单模型，分别改写成 Kotlin 和 Rust 版本，明确处理优惠码可空和金额非负约束。
- `迁移题`：如果你来自 Python 或 JavaScript，写一页短文解释为什么 Kotlin / Swift / Rust 会强迫你更早定义空值语义。
- `评审题`：审查一个 `status: String`、`amount: Double`、`coupon: String?` 的订单模型，指出至少 3 个会在真实业务里出问题的地方。
- `变更实验`：给订单增加 `退款中`、`部分退款` 两个状态，比较动态语言和强约束语言里，哪些修改会更早暴露不完整建模。

## 第 2 章：数据的容器

- `实现题`：分别用 Python、Go、Rust 写“按用户分组统计已支付订单总额”，并说明为什么选列表、映射或集合。
- `迁移题`：从 Java 切到 Go，解释为什么 Go 的内建 `map` 与切片足够朴素，却依然适合大多数服务端代码。
- `评审题`：看到一个接口把内部 `HashMap` 直接暴露给调用方时，你会担心哪些共享修改风险？
- `变更实验`：在原有“按用户统计”需求上加入“保留原始下单顺序”和“去重订单号”，看看容器选型如何变化。

## 第 3 章：逻辑的流转

- `实现题`：把一个三层嵌套的下单校验流程，重写成守卫式返回版本。
- `迁移题`：从 Python 或 Go 切到 Rust / Kotlin / Swift，说明模式匹配和穷尽性检查会在哪些业务场景明显改变你的写法。
- `评审题`：审查一个带巨大 `default` 分支的状态机实现，指出为什么这会让后续需求变更难以及时暴露漏洞。
- `变更实验`：给订单状态流转新增 `风控审核中`，比较不同语言里哪些位置会自动提醒你漏改了逻辑。

## 第 4 章：行为的封装

- `实现题`：实现一个“折扣策略可插拔”的结算函数，再实现一个带缓存的闭包版本。
- `迁移题`：如果你来自 Java，解释为什么第一次看到 JavaScript / Python 的函数一等公民会觉得轻松；如果你来自 Python，解释第一次看到 Rust 的 `Fn / FnMut / FnOnce` 为什么会不适。
- `评审题`：审查一个异步闭包直接捕获整个 `OrderService` 对象的实现，说明它会带来哪些生命周期和测试问题。
- `变更实验`：把“普通折扣”和“VIP 折扣”改成运行时策略切换，比较函数参数、对象策略、闭包捕获三种方案的优缺点。

## 第 5 章：世界的抽象

- `实现题`：把一个直接在支付类里写日志、重试、审计的实现，重构成组合式结构。
- `迁移题`：从 Java 切到 Go，解释为什么“小接口由使用方定义”会改变你的抽象习惯。
- `评审题`：看到一个 15 个方法的大接口时，你会怎样判断它是在隔离变化，还是在制造概念负担？
- `变更实验`：给支付流程新增“限流”和“灰度回滚”，比较继承式和组合式方案在改动波及上的差异。

## 第 6 章：内存的哲学

- `实现题`：分别用 Go、Swift、Rust 写一个订单导出功能，明确展示资源申请与释放边界。
- `迁移题`：从 Java / Python 切到 Rust 时，写出你最容易带错的 3 个生命周期直觉。
- `评审题`：审查一段把大对象放进全局缓存、又被闭包捕获的代码，指出它为什么会让对象“活得太久”。
- `变更实验`：给订单详情页加入长生命周期监听器和图片缓存，分析 GC、ARC、Ownership 下分别最容易出什么问题。

## 第 7 章：并发的艺术

- `实现题`：实现“同时拉取订单、库存、价格并在任一分支失败时中止整体”的逻辑，至少用两门语言完成。
- `迁移题`：比较 JavaScript 的 `Promise.all`、Go 的 goroutine、Kotlin 的 `coroutineScope`，说明它们在取消和任务边界上的差异。
- `评审题`：审查一个大量启动 goroutine 但没有 `context` 取消的实现，说明为什么它会变成线上隐患。
- `变更实验`：把原本只需要并发等待的场景，改成包含 CPU 密集计算，分析原有并发模型会在哪里失效。

## 第 8 章：异常的妥协

- `实现题`：为“提交订单”设计一套错误分层，至少区分领域错误、技术错误和边界错误。
- `迁移题`：从异常语言切到 Go 或 Rust，写出你最不习惯的 3 个地方，以及这些不习惯背后其实在提醒你什么。
- `评审题`：审查一个把所有错误都包成 `RuntimeException("pay failed")` 的实现，指出这会损失哪些排障和治理能力。
- `变更实验`：新增“支付网关超时自动重试，但库存不足不重试”的规则，比较不同错误模型下代码和架构的变化。

## 第 9 章：看懂第九门语言

- `实现题`：选一门你没深入用过的语言，用本书的七问法写一份 1 到 2 页的迁移评估。
- `迁移题`：回顾你最熟悉的语言，列出其中 3 个你曾误以为是“常识”，其实只是那门语言默认假设的地方。
- `评审题`：假设团队准备引入一门新语言，请写一份评审清单，至少覆盖类型、并发、生命周期、错误处理和工程治理。
- `变更实验`：用同一个小业务题分别在两门陌生语言里实现，然后写一份复盘：真正改变你写法的不是语法，而是哪 3 条底层制度。

---

## 进阶使用建议

- 如果是个人阅读：每章做 1 道实现题 + 1 道迁移题
- 如果是团队读书会：每章做 1 道评审题 + 1 道变更实验题
- 如果是架构训练：优先做第 5、6、7、8、9 章的迁移题和评审题

真正有价值的，不是把答案写出来一次，而是让你以后遇到新语言时，仍然知道该从哪里开始问问题。
